00

Start hereread this box and the whole handbook clicks into place

If you're moving from infrastructure and networking into Kubernetes, the honest truth is: you don't need to become a Docker expert first. You need a specific, surprisingly small slice of Docker, and then you spend most of your energy on Kubernetes itself. This handbook is built to give you that slice and nothing extra.

◆ The one idea everything hangs on

Docker is a packaging tool. Kubernetes is a running tool.

Docker's job finishes the moment you have an image — a sealed, portable box containing your app and everything it needs. Kubernetes' job starts there: it takes that box and runs many copies of it, restarts them when they die, spreads them across machines, and wires up networking. They are not rivals. They are two links in one chain.

You use Docker to build the box. You use Kubernetes to run an army of those boxes.

Here's the exact path this handbook follows. It's one small app — a tiny Node.js API — carried the whole way, so nothing ever feels disconnected:

By the end you'll be able to answer, with a real example in hand, every question you came in with: how a Docker image is built, what multi-stage builds are and whether Kubernetes uses them, how Compose differs from Kubernetes, what a manifest means line by line, and how to actually get it running on a single AWS EC2 box.

01

Docker vs Kubernetesthe difference, in plain terms first

✎ Analogy first

Docker is the machine that packs your goods into a standard shipping container and seals it. Kubernetes is the port: cranes, schedules, spare containers ready to swap in, re-routing when a crane breaks. One makes the container; the other operates a whole yard of them.

People treat "Docker vs Kubernetes" like a versus match, but they solve different problems. Docker answers "how do I package this app so it runs the same everywhere?" Kubernetes answers "how do I run hundreds of those packages reliably, at scale, without babysitting them?"

You can absolutely use Docker without Kubernetes (a single container on your laptop). And Kubernetes runs the containers that Docker-style tooling produced. The overlap is thin: Docker has a tiny built-in orchestrator called Swarm, and that's the only place the two genuinely compete — and Swarm lost. For real orchestration, the industry standardized on Kubernetes.

Side-by-side responsibilities

ConcernDocker handlesKubernetes handles
Packaging an app✓ builds the image (app + deps + OS libs)— consumes the image
Running one container✓ docker run on one host✓ but as a managed Pod
Running many copies△ manual / Swarm only✓ core feature (replicas)
Restart on crash△ restart policy, single host✓ self-healing across the cluster
Spread across machines✗✓ the scheduler places pods on nodes
Rolling updates / rollback△ Swarm only✓ built into Deployment
Service discovery / load balance△ basic DNS on one host✓ Service + cluster DNS
Autoscaling✗✓ HPA on CPU/metrics
Config & secrets△ env flags, files✓ ConfigMap / Secret
Scopeone machinea fleet of machines as one system

Notice the pattern: nearly every Docker strength is about a single machine, and nearly every Kubernetes strength is about coordinating many machines and many copies. That's the whole distinction in a sentence.

→ How this shows up later

When you write a Kubernetes Deployment in Part 09, the fields you'll spend the most time on — replicas, selector, probes, resources — are precisely the "many copies, keep them healthy, place them well" concerns from the right-hand column. Docker never had to think about those, which is exactly why Kubernetes exists.

02

Why we still use Docker in a Kubernetes worldincluding the "Kubernetes dropped Docker!" confusion

This is the question that trips up almost everyone entering Kubernetes, partly because of a genuinely confusing headline from a few years back. Let's clear it completely.

The "Kubernetes removed Docker" myth

In 2020, Kubernetes announced it was deprecating something called dockershim, and headlines screamed that "Kubernetes is dropping Docker." That was misleading. Here's what actually happened, plainly:

  • Kubernetes doesn't run containers itself — it delegates to a container runtime (the low-level thing that actually starts a container from an image).
  • Docker was never designed to be that low-level runtime for Kubernetes, so Kubernetes shipped a little adapter called dockershim to talk to Docker anyway. That adapter was extra maintenance baggage.
  • Kubernetes removed the adapter and now talks directly to lightweight runtimes like containerd (which, ironically, is the very same runtime that lives inside Docker) or CRI-O.
◆ The part that actually matters to you

What got removed was a runtime adapter, not the thing you care about. Images built with Docker follow the open OCI standard, and they run perfectly on Kubernetes regardless of runtime. Your docker build workflow is completely unaffected.

Kubernetes stopped using Docker as an engine. It never stopped using the boxes Docker builds.

So where do you still use Docker, day to day?

Even on a pure Kubernetes team, Docker (or a Docker-compatible tool) shows up constantly:

◆ Building images
  • You still write a Dockerfile.
  • You still run docker build in CI to produce the image K8s will run.
  • Multi-stage builds (Part 06) are still how you keep images small.
◆ Local development
  • docker run to smoke-test the image before it ever touches a cluster.
  • Compose to spin up your app + database locally without Kubernetes.
◆ Debugging & registries
  • docker push / pull to registries (Docker Hub, ECR).
  • docker inspect, logs, exec to poke at a container in isolation.

In other words: Kubernetes is where images go to run; Docker is where images come from. On the job you'll build with Docker in the morning and deploy to Kubernetes in the afternoon, every day. That's why learning Docker first isn't wasted — it's the on-ramp.

▲ Nuance worth knowing

Some teams have swapped the build tool too — using BuildKit, Buildah, Kaniko, or Podman instead of the Docker CLI (Kaniko in particular builds images inside Kubernetes without a Docker daemon). But these all read the same Dockerfile and produce the same OCI images. Learn Docker's model and these are a small step sideways, not a new mountain.

Where Docker actually fits inside a Kubernetes workflow

Let's get concrete about the exact moments Docker shows up once your app lives on Kubernetes. The clean way to hold it in your head: Docker owns everything up to the image; Kubernetes owns everything from the image onward. That single handoff is the whole relationship — and it happens many times a day.

Now the specific scenarios. In every row below, notice that the Kubernetes column is just context — the real action, the thing you actually type, is Docker:

Scenario / use caseExactly how Docker fits inWhat you actually run
Shipping a new version (the CI/CD pipeline) Kubernetes can't build anything — it only runs images. So every deploy begins with Docker turning your code into the image the cluster will later pull. docker build → docker push
Sanity-checking before a rollout Before wasting a cluster deploy, you run the exact image locally to confirm it boots, serves traffic, and reads its env vars. Catches "won't even start" bugs instantly. docker run -p … image:tag
Debugging a crashing pod (CrashLoopBackOff) When a pod keeps restarting, pull the same image and run it on your laptop with the same env, then shell in and poke around — no cluster noise, full interactive control. docker run -it … sh · docker logs
Getting the image to the cluster The cluster's nodes can't see your machine. Docker tags and uploads the image to a registry (ECR / Hub) that Kubernetes can pull from. This is the literal bridge. docker tag · docker push
Keeping images small & safe Smaller images = faster pod starts and autoscaling in the cluster. You inspect layers and slim them with Docker before anything is deployed. docker history · docker image inspect
Golden / base images for the team Platform teams maintain a hardened base image with Docker; every app image inherits FROM it. K8s just runs whatever comes out. docker build the shared base
Everyday local development Instead of spinning up a whole cluster to code, you run the app plus its database with Compose on your machine, then deploy to K8s only when it's ready. docker compose up
◆ The pattern under all seven rows

Every scenario is Docker doing build-time work — creating, testing, inspecting, or shipping an image — and Kubernetes only entering once that image exists. You will never "use Docker" and "use Kubernetes" at the same instant; you hand off from one to the other at the registry, over and over.

Docker is the workshop. Kubernetes is the stage. Nothing reaches the stage that wasn't first built in the workshop.

▲ So is Docker "inside" Kubernetes?

No — and that's the subtlety worth keeping straight. Docker isn't a component running within your cluster. It sits alongside Kubernetes in your workflow: on your laptop and in your CI runners, producing the images. The cluster's own low-level runtime (containerd/CRI-O) is what actually launches containers from those images. Docker's role is upstream of Kubernetes, not part of it.

03

How much Docker do you actually need?the 20% that carries you into Kubernetes

Here's the reassuring part. Docker has a huge surface area — Swarm, macvlan networking, plugins, contexts, image squashing — but for the goal of "get good at Kubernetes," most of it is noise. Below is the honest split: learn the left column well, skim the middle, and skip the right entirely until you have a reason.

◆ Learn well (the core)
  • What an image vs a container is
  • Reading & writing a Dockerfile
  • Layers & build caching
  • build, run, ps, logs, exec
  • Tag & push to a registry
  • Ports (-p), env vars (-e)
  • Volumes (persist data)
  • Multi-stage builds (small images)
  • Basics of Compose
◈ Skim (nice, not urgent)
  • User-defined networks & container DNS
  • docker inspect with Go templates
  • Restart policies & healthchecks
  • Resource limits (--cpus, --memory)
  • .dockerignore & BuildKit cache mounts
  • Non-root users & image scanning (Trivy)
◇ Skip for now
  • Docker Swarm (Kubernetes replaces it)
  • macvlan / advanced net drivers
  • Docker plugins & contexts
  • Image squashing / DockerSlim
  • Custom base images from scratch
  • Swarm secrets, stacks, routing mesh
✎ Rule of thumb

If a Docker feature is about running things across many machines — that's Kubernetes' job now, so skip it in Docker. If it's about building or testing a single image — learn it, because that's the part Kubernetes hands back to you.

Realistically: a focused engineer can get comfortable with the "learn well" column in a weekend or two of hands-on practice, then move straight into Kubernetes. Everything in this handbook lives in that left column, plus the parts of the middle column that make your images production-ready for a cluster.

04

Building a Docker imagethe process, and every Dockerfile instruction that matters

Before the how, lock in the two nouns you'll say a thousand times:

TermWhat it isAnalogy
ImageA read-only, layered template: your app + runtime + libraries + OS files, frozen together. It doesn't run — it's a blueprint on disk.A sealed, labelled shipping box (or a class in code).
ContainerA running instance of an image, with a thin writable layer on top. You can start many containers from one image.The box actually opened and doing work (an object created from the class).

Images are made of layers

An image isn't one blob — it's a stack of layers, and each line in your Dockerfile that changes the filesystem creates one. Layers are cached and shared: if ten images all start FROM node:20, that base layer is stored once. This is why the order of your Dockerfile lines matters — put the things that rarely change near the top so their layers stay cached, and the things that change often (your source code) near the bottom.

The Dockerfile instructions you'll actually use

A Dockerfile is just a recipe read top-to-bottom. These are the instructions worth memorising — the rest you can look up:

InstructionWhat it doesNote for later
FROMSets the base image every other layer builds on. Always the first real line.Smaller base = smaller image = faster pods.
WORKDIRSets the working directory inside the image (and creates it). All later commands run from here.Use it instead of cd.
COPYCopies files from your project into the image.Copy dependency files first, source last (caching).
ADDLike COPY but can fetch URLs and auto-extract tarballs.Prefer COPY; only use ADD when you need those extras.
RUNExecutes a command at build time and bakes the result into a layer (e.g. install packages).Chain with && to keep layers lean.
ENVSets an environment variable baked into the image.K8s can override these at runtime.
ARGA build-time variable you can pass with --build-arg.Gone after build; not visible at runtime.
EXPOSEDocuments which port the app listens on. Purely informational.Does not publish the port — that's -p or a K8s Service.
USERSwitches to a non-root user for later steps and at runtime.Security best practice K8s expects.
CMDThe default command run when a container starts. Overridable.One per Dockerfile (last wins).
ENTRYPOINTThe fixed executable; CMD becomes its default arguments.Use for "this image IS this program".
▲ CMD vs ENTRYPOINT — the classic confusion

ENTRYPOINT is the program that always runs; CMD is the default arguments to it (and is easy to override). A common pattern: ENTRYPOINT ["node"] + CMD ["app.js"]. If you only need "run this by default, but let me override the whole thing," plain CMD ["node","app.js"] is simpler. For most app images, just use CMD.

The build → tag → push lifecycle

Producing an image and getting it somewhere Kubernetes can reach it is a three-move dance you'll repeat forever:

→ Why the registry step is non-negotiable for K8s

On your laptop, a freshly built image is right there. But a Kubernetes cluster is other machines — they can't see your laptop's local images. So the image must live in a registry the cluster can pull from. That's why "tag + push" is the bridge between Docker and Kubernetes, and why your Deployment's image: field points at a registry path, not a local name.

05

A single container, end to endthe smallest complete example — build it, run it, ship it

Let's make it real. Our app is a tiny Node.js API called meditrack-api. Two files, no framework magic. This same app rides through every remaining part of the handbook.

1 · The app

app.jsjavascript
const express = require('express');
const app = express();

// Read config from the environment. Defaults let it run anywhere,
// but Compose and Kubernetes will override these later.
const PORT = process.env.PORT || 3000;
const GREETING = process.env.GREETING || 'Hello from your laptop';
const VERSION = process.env.APP_VERSION || 'dev';

app.get('/', (req, res) => {
  res.json({ service: 'meditrack-api', message: GREETING, version: VERSION });
});

// A tiny health endpoint — Kubernetes probes will hit this in Part 09.
app.get('/healthz', (req, res) => res.status(200).send('ok'));

// Bind to 0.0.0.0, NOT localhost — see the gotcha below.
app.listen(PORT, '0.0.0.0', () => console.log(`listening on ${PORT}`));
package.jsonjson
{
  "name": "meditrack-api",
  "version": "1.0.0",
  "main": "app.js",
  "scripts": { "start": "node app.js" },
  "dependencies": { "express": "^4.19.2" }
}
▲ The #1 "works on my machine, not in the container" bug

If your app listens on localhost (127.0.0.1) inside a container, nothing outside the container can reach it — not -p, not a Kubernetes Service. Inside a container, always bind to 0.0.0.0 (all interfaces). This one line saves hours of "the port is mapped but curl hangs" debugging.

2 · The Dockerfile (naive first version)

We'll start with the obvious version so the concepts are clear. Part 06 rewrites it properly.

Dockerfiledockerfile
FROM node:20                 # base image: Node 20 on a full OS
WORKDIR /app                  # work inside /app from here on
COPY package*.json ./         # copy dep manifest FIRST (cache-friendly)
RUN npm install               # install deps as a cached layer
COPY . .                      # copy the rest of the source
EXPOSE 3000                   # document the port (informational)
CMD ["node", "app.js"]      # default command when the container starts

See why package*.json is copied before the source? If you only change app.js, Docker reuses the cached npm install layer and the rebuild is instant. Copy everything at once and every code change re-installs all dependencies. Small ordering choice, big speed difference.

4 · Run it

terminalbash

# -e sets an env var, --name gives it a friendly handle

◆ Read -p 8080:3000 left-to-right

The left number is the port on your host machine; the right is the port inside the container. So 8080:3000 means "traffic to my laptop's 8080 goes to the container's 3000." Kubernetes splits this same idea across a Service (port) and a container (targetPort) — you'll spot the parallel in Part 09.

5 · Tag & push (so a cluster can pull it)

To hand the image to Kubernetes later, it needs to live in a registry. Two common targets — Docker Hub and AWS ECR:

06

Multi-stage buildstiny images — and whether Kubernetes uses them

The naive Dockerfile in Part 05 works, but it's fat. It ships the entire Node build toolchain, dev dependencies, and a full OS — often 1 GB+ for a tiny app. Multi-stage builds fix this, and understanding them is one of the highest-leverage Docker skills for Kubernetes.

✎ Analogy first

Think of a big messy kitchen where you cook the meal (all the pots, flour everywhere, the mixer). When it's done, you don't ship the kitchen — you plate just the meal onto a clean dish and send that. Multi-stage builds cook in a "builder" stage, then copy only the finished result into a tiny clean final image.

How it works

A multi-stage Dockerfile has more than one FROM. Each FROM starts a fresh stage. You do the heavy work (compiling, installing) in an early stage, then COPY --from=<stage> only the artifacts you actually need into a small final stage. Everything left behind — compilers, caches, dev deps — never ships.

Dockerfile (multi-stage)dockerfile
# ---------- Stage 1: BUILD (the messy kitchen) ----------
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev          # clean, reproducible, prod-only deps
COPY . .

# ---------- Stage 2: RUNTIME (the clean plate) ----------
FROM node:20-slim AS runtime
ENV NODE_ENV=production
WORKDIR /app
RUN groupadd -r app && useradd -r -g app app   # a non-root user
COPY --from=build /app/node_modules ./node_modules
COPY --from=build /app/app.js ./app.js
COPY --from=build /app/package.json ./package.json
USER app                        # run unprivileged
EXPOSE 3000
CMD ["node", "app.js"]
Naive single-stageMulti-stage
Contentsfull OS + Node + build tools + dev deps + appslim OS + Node + prod deps + app
Typical size~1 GB~150–200 MB (distroless: even less)
Attack surfacelarge (compilers, shells, extra packages)small (fewer things to exploit)
Runs asroot (default)non-root user

Do we use multi-stage in Kubernetes? And how?

→ The direct answer

Multi-stage is a build-time technique — it happens entirely inside docker build, before Kubernetes is ever involved. Kubernetes never sees your stages. It only pulls and runs the final image. So there's no "multi-stage setting" in a manifest — it's not a Kubernetes feature at all.

Kubernetes doesn't know or care how the box was packed — only what's finally inside it.

But yes, you absolutely use multi-stage builds when your target is Kubernetes — arguably more than anywhere else. Here's the "how," meaning how it pays off in a cluster:

  • Faster pod startup & autoscaling. When the HPA adds pods or a node reboots, each new pod must pull the image first. A 180 MB image starts far faster than a 1 GB one — the difference between scaling in seconds vs minutes under load.
  • Cheaper, faster everything. Smaller images mean less registry storage, less network transfer on every rollout, and less disk on every node.
  • Smaller attack surface. Clusters run untrusted-ish workloads at scale; a slim, non-root image with no shell or compiler is a much harder target. Security teams love it.
  • Cleaner rollouts. Rolling updates pull the new image onto every node; slim images make updates quick and low-risk.

So the mental model is: you choose multi-stage because the image is going to Kubernetes, even though Kubernetes itself is oblivious to the technique. You build the small image with Docker; the Deployment just references it in image:.

07

Docker Composemany containers at once — your local dev stand-in for a cluster

Real apps aren't one container — they're an app plus a database, a cache, maybe a queue. Running each with a long docker run line is painful. Compose lets you declare the whole set in one YAML file and bring it all up with a single command. This is your first taste of "describe the desired state in YAML," which is the exact mindset Kubernetes runs on.

docker-compose.ymlyaml
services:
  api:                              # our Node app
    build: .                        # build from the Dockerfile here
    image: meditrack-api:1.0.0
    ports:
      - "3000:3000"               # host:container
    environment:
      GREETING: "Hello from Compose"
      APP_VERSION: "1.0.0"
      DATABASE_URL: "postgres://meditrack:secret@db:5432/meditrack"
      REDIS_URL: "redis://cache:6379"
    depends_on: [db, cache]         # start db & cache first
    restart: unless-stopped

  db:                               # PostgreSQL
    image: postgres:16
    environment:
      POSTGRES_USER: meditrack
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: meditrack
    volumes:
      - pgdata:/var/lib/postgresql/data   # persist DB across restarts
    restart: unless-stopped

  cache:                            # Redis
    image: redis:7
    restart: unless-stopped

volumes:
  pgdata:                           # a named volume Docker manages
◆ The magic you get for free: DNS by service name

Compose puts all three containers on one network and lets them find each other by service name. That's why the app's DATABASE_URL says @db:5432 — db is literally the hostname of the Postgres container. No IPs, no config. Kubernetes does the identical thing with Services and cluster DNS, so this habit transfers directly.

The handful of Compose commands

terminalbash
$ docker compose up -d --build    # build + start everything in background
$ docker compose ps               # what's running
$ docker compose logs -f api       # follow one service's logs
$ docker compose exec api sh       # shell into the api container
$ docker compose down              # stop & remove (add -v to wipe volumes)

Compose vs Kubernetes — same idea, different scale

Compose and Kubernetes are often confused because both use YAML to describe multi-container apps. The difference is where they run and what they guarantee:

Docker ComposeKubernetes
Runs onone machine (usually your laptop)a cluster of machines
Best forlocal dev, quick demos, small single-host appsproduction, scale, high availability
Self-healingbasic restart policyfull — reschedules pods, replaces nodes
Scalingmanual (--scale), one hostdeclarative + automatic (HPA), many hosts
Rolling updatesno real strategybuilt in, with rollback
Config fileone docker-compose.ymlseveral manifests (Deployment, Service, …)
✎ How teams actually use both

Compose for "run the whole app on my machine while I code." Kubernetes for "run it for real in production." Same app, two front doors. Learning Compose first makes Kubernetes' YAML feel familiar instead of foreign.

08

From Compose to Kubernetes manifeststranslate what you already understand

You now have a working docker-compose.yml. The best way to learn Kubernetes manifests is to see each Compose concept map onto its Kubernetes equivalent. Nothing here is brand new — it's the same ideas, unbundled into separate objects so a cluster can manage each independently.

In Compose you wrote…In Kubernetes it becomes…Why it splits apart
a service: with an imageDeployment (which creates Pods)K8s needs to manage many replicas, not one container.
ports: "3000:3000"Service (+ Ingress for public URLs)A stable network address is decoupled from the pods behind it.
environment: (plain values)ConfigMapConfig lives outside the image and can change without a rebuild.
environment: (passwords)SecretSensitive values get separate, restricted handling.
volumes:PersistentVolumeClaimStorage is requested abstractly; the cluster provisions it.
depends_on:(no exact match) — use probes + init containersK8s prefers "keep retrying until healthy" over strict ordering.
restart: unless-stoppedautomatic — the Deployment always keeps pods aliveSelf-healing is the default, not an opt-in flag.
--scale api=3replicas: 3 (+ HPA)Desired count is declared; K8s makes reality match.
◆ The mental shift: imperative → declarative

With Docker you mostly give commands ("run this container now"). With Kubernetes you declare a desired state ("I want 3 healthy replicas of this image, reachable on port 80") and a control loop constantly works to make the cluster match it. If a pod dies, you didn't ask K8s to restart it — it simply notices reality drifted from your declaration and fixes it.

✎ Shortcut worth knowing (don't lean on it)

There's a tool called kompose that auto-converts a docker-compose.yml into rough Kubernetes manifests. Great for a first draft or to see the mapping — but the output always needs hand-tuning. Learn to read and write the YAML yourself (next part), and treat kompose as a scaffold, not a crutch.

Places this object in our drawer from step 1. Without it, the object lands in default.
data:
The actual key/value pairs. Everything here is plain text — never put passwords in a ConfigMap.
  GREETING: "..."
A value the Deployment will inject into the container as the GREETING env var — the exact one app.js reads.

3 · Secret — the same idea, for sensitive values

A Secret is structurally like a ConfigMap but meant for passwords, tokens, and connection strings. Kubernetes handles it with extra care (restricted access, kept out of plain logs).

secret.yamlyaml
apiVersion: v1
kind: Secret
metadata:
  name: meditrack-secret
  namespace: meditrack
type: Opaque
stringData:
  DATABASE_URL: "postgres://meditrack:secret@db:5432/meditrack"
secret.yamlline by line
the line
what it means
kind: Secret
Same shape as a ConfigMap, but flagged as sensitive so Kubernetes treats access more strictly.
type: Opaque
The generic "just some key/value secret" type. (Other types exist for things like registry logins and TLS certs.)
stringData:
Lets you write values as plain readable text. Kubernetes base64-encodes them for you on the way in. Use stringData while learning — it's far friendlier than hand-encoding.
  DATABASE_URL: "..."
The connection string. The Deployment pulls it in as an env var — but sourced from the Secret, not the ConfigMap.
▲ Secrets are only lightly protected by default

Out of the box a Kubernetes Secret is merely base64-encoded (trivially decodable), not truly encrypted. For real environments you'd enable encryption-at-rest, tight RBAC, or an external vault. For learning, the pattern above is exactly right — just don't mistake base64 for security.

4 · Deployment — the heart of it all

This is the manifest you'll live in. It says: "run N healthy copies of this image, keep them alive, and here's exactly how each copy should be configured." Every Docker concept you learned reappears here as a field.

deployment.yamlyaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: meditrack-api
  namespace: meditrack
  labels:
    app: meditrack-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: meditrack-api
  template:
    metadata:
      labels:
        app: meditrack-api
    spec:
      containers:
        - name: api
          image: <your-registry>/meditrack-api:1.0.0
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 3000
          envFrom:
            - configMapRef:
                name: meditrack-config
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: meditrack-secret
                  key: DATABASE_URL
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"
          readinessProbe:
            httpGet:
              path: /healthz
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /healthz
              port: 3000
            initialDelaySeconds: 10
            periodSeconds: 15
deployment.yamlevery line explained
the line
what it means & why it's there
apiVersion: apps/v1
Deployments live in the apps API group (unlike core objects like Namespace/Service which are plain v1). If you ever get "no matches for kind Deployment", a wrong apiVersion is usually why.
kind: Deployment
The controller that manages a set of identical Pods for you — creating, replacing, updating, and scaling them.
metadata:   name: meditrack-api   namespace: meditrack
Names the Deployment and files it in our namespace. This name is what you'll type in kubectl commands.
  labels:     app: meditrack-api
A label on the Deployment object itself — handy for filtering (kubectl get deploy -l app=meditrack-api). Labels are just key/value tags used everywhere in K8s.
spec:
The desired state begins here. Everything below describes what "correct" looks like; the cluster continuously works to match it.
  replicas: 3
Run three identical Pods. Kill one and the Deployment immediately starts a replacement to get back to three. This single number is the "self-healing + scaling" you came to Kubernetes for.
  selector:     matchLabels:       app: meditrack-api
How the Deployment knows which Pods are "its own." It looks for Pods carrying the label app: meditrack-api. This must match the template labels just below — that's the contract.
  template:
The blueprint for each Pod. Everything under here is "stamp out one of these, three times." Change the template → the Deployment rolls out new Pods.
    metadata:       labels:         app: meditrack-api
The label stamped on every Pod this creates. It must equal the selector.matchLabels above, or the Deployment won't recognise its own Pods. It's also how the Service (step 5) finds them.
    spec:       containers:
The Pod's spec. A Pod can hold more than one container, so this is a list — hence the - on the next line.
      - name: api
A name for this container inside the Pod. Used in commands like kubectl logs <pod> -c api.
        image: <registry>/meditrack-api:1.0.0
The exact image to run — the one you pushed in Part 05. This is the literal handoff point from Docker to Kubernetes. Pin a real version tag, never :latest, so rollouts are predictable.
        imagePullPolicy: IfNotPresent
Pull the image only if the node doesn't already have it. (Always re-pulls every time — useful with mutable tags; IfNotPresent is fine for pinned versions.)
        ports:         - containerPort: 3000
Declares the port the app listens on inside the container — the same 3000 from app.js. Mostly documentation; the Service is what actually routes to it.
        envFrom:         - configMapRef:             name: meditrack-config
Inject every key from the ConfigMap as an env var. So GREETING and APP_VERSION appear inside the container automatically — that's how config reaches your code without a rebuild.
        env:         - name: DATABASE_URL           valueFrom:             secretKeyRef:               name: meditrack-secret               key: DATABASE_URL
Pull one specific value out of the Secret and expose it as the DATABASE_URL env var. Same idea as envFrom, but for a single sensitive value sourced from a Secret instead of a ConfigMap.
        resources:           requests:             cpu: "100m"             memory: "128Mi"
requests = the minimum the scheduler reserves for this Pod when deciding which node it fits on. 100m = 0.1 of a CPU core; 128Mi ≈ 128 MB. Think of it as the Pod's guaranteed floor.
          limits:             cpu: "500m"             memory: "256Mi"
limits = the ceiling. Exceed the memory limit and the container is killed (OOMKilled); exceed CPU and it's throttled. Setting both protects noisy neighbours from starving the node.
        readinessProbe:           httpGet:             path: /healthz             port: 3000           initialDelaySeconds: 5           periodSeconds: 10
"Is this Pod ready for traffic yet?" Kubernetes hits /healthz every 10s (after a 5s grace). Until it returns 200, the Service won't send requests to this Pod. This is why app.js had a /healthz endpoint. Prevents traffic hitting a still-booting Pod.
        livenessProbe:           httpGet:             path: /healthz             port: 3000           initialDelaySeconds: 10           periodSeconds: 15
"Is this Pod still alive, or wedged?" If /healthz stops responding, Kubernetes restarts the container. Readiness controls traffic; liveness controls restarts. Same endpoint, two different jobs.
◆ The label handshake — the thing beginners miss

Three label spots must agree: the Deployment's selector.matchLabels, the Pod template's labels, and (next) the Service's selector. That shared label app: meditrack-api is the invisible wiring that lets the Deployment own its Pods and the Service find them. Get these out of sync and you'll see "0 pods" or a Service that routes to nothing.

5 · Service — a stable address for the Pods

Pods come and go (and each gets a new IP). A Service is a fixed front door: one stable name and IP that always load-balances to whichever Pods currently match its label selector.

service.yamlyaml
apiVersion: v1
kind: Service
metadata:
  name: meditrack-api
  namespace: meditrack
spec:
  type: ClusterIP
  selector:
    app: meditrack-api
  ports:
    - port: 80
      targetPort: 3000
service.yamlline by line
the line
what it means
kind: Service
A stable networking endpoint that fronts a changing set of Pods and load-balances across them.
  type: ClusterIP
The default type: reachable only inside the cluster. Perfect for internal traffic. (We'll switch this to NodePort in Part 10 to reach it from a browser on EC2.)
  selector:     app: meditrack-api
Which Pods to send traffic to. The same label handshake again — it targets every Pod tagged app: meditrack-api, i.e. the ones our Deployment made.
  ports:   - port: 80     targetPort: 3000
port = the port the Service listens on (what other things call). targetPort = the container's port it forwards to. This is Docker's -p 80:3000 split into two named fields.

6 · HorizontalPodAutoscaler — grow and shrink automatically

The capability Docker simply doesn't have. This watches CPU and changes replicas for you between a floor and a ceiling.

hpa.yamlyaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: meditrack-api
  namespace: meditrack
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: meditrack-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
hpa.yamlline by line
the line
what it means
  scaleTargetRef:     kind: Deployment     name: meditrack-api
Which workload to scale. It points straight at our Deployment by name — the HPA drives that Deployment's replica count up and down.
  minReplicas: 2   maxReplicas: 10
Never fewer than 2 (availability), never more than 10 (cost/ceiling). The HPA moves within this band.
  metrics: … cpu   averageUtilization: 70
The trigger. Aim to keep average CPU near 70%. Above it → add Pods; well below → remove them. Requires the metrics-server add-on to actually read CPU.

Order matters a little: create the namespace and the config/secret before the things that reference them.

10

Deploy on a single EC2 instancefrom YAML on your laptop to a live URL on AWS

"Once I have the YAML, how do I actually run this on one EC2 box?" There are two honest answers depending on which YAML you mean, and it's worth doing both because they teach different halves of the picture.

Path A — Compose on EC2 (the "run it as Docker" answer)

If by "the YAML" you mean your docker-compose.yml, this is the fastest way to get the whole app (api + db + cache) live on one box.

  1. Launch the instance

    An Ubuntu EC2 (t3.small or bigger). In its security group, allow inbound 22 (SSH, your IP only) and 3000 (the app). This SG step is the single most common reason "it works on the box but not in my browser."

  2. SSH in

    terminalbash
    $ ssh -i key.pem ubuntu@<EC2_PUBLIC_IP>

  3. Install Docker Engine + the Compose plugin

    terminalbash
    $ curl -fsSL https://get.docker.com | sudo sh
$ sudo usermod -aG docker ubuntu && newgrp docker   # run docker without sudo
$ docker --version && docker compose version

    tip The convenience script above installs Engine, CLI, containerd, and the Compose plugin in one shot — ideal for a learning box.

  4. Get your files onto the box & bring it up

    terminalbash
    $ git clone <your-repo> meditrack && cd meditrack
$ docker compose up -d --build
$ docker compose ps                 # all three should be "running"
$ curl http://localhost:3000        # works ON the box

  5. Open it from your browser

    Visit http://<EC2_PUBLIC_IP>:3000. If curl worked on the box but the browser hangs, it's almost always the security group — re-check that port 3000 is open inbound.

Path B — Run the Kubernetes manifests on one EC2 (using k3s)

If by "the YAML" you mean the manifests from Part 09, you need an actual (tiny) Kubernetes cluster. k3s is a full, lightweight Kubernetes that installs as a single node in one command — perfect for a single EC2.

  1. Prep the instance

    Ubuntu EC2 (t3.medium recommended — a cluster wants a bit more RAM). Security group: allow 22 and 30080 (our NodePort, chosen below).

  2. Install k3s (bundles Kubernetes + containerd + kubectl)

    terminalbash
    $ curl -sfL https://get.k3s.io | sh -
$ sudo k3s kubectl get nodes         # should show one Ready node

# let plain "kubectl" work without sudo:
$ mkdir -p ~/.kube
$ sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
$ sudo chown $(id -u):$(id -g) ~/.kube/config
$ kubectl get nodes

    note k3s uses its own bundled containerd as the runtime — a live reminder that the cluster runs images without Docker itself, exactly as Part 02 described.

  3. Make the image reachable

    The cluster pulls from a registry, so push your image to Docker Hub or ECR (Part 05) and set the Deployment's image: to that path. For a public image nothing more is needed; for a private one, add an image-pull secret.

    terminal — alt: import a local image into k3sbash
    # if you built the image on the EC2 box itself and don't want a registry:
$ docker save meditrack-api:1.0.0 | sudo k3s ctr images import -

  4. Use a NodePort Service so the browser can reach it

    Swap the Service type to NodePort and pin a port. NodePort opens that port on the EC2's own IP — the simplest public access on a single node (no cloud load balancer needed).

    service-nodeport.yamlyaml
    apiVersion: v1
kind: Service
metadata:
  name: meditrack-api
  namespace: meditrack
spec:
  type: NodePort            # exposes a port on the node's IP
  selector:
    app: meditrack-api      # same label handshake
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30080       # reachable at <EC2_IP>:30080

  5. Apply everything & verify

    terminalbash
    $ kubectl apply -f namespace.yaml -f configmap.yaml -f secret.yaml
$ kubectl apply -f deployment.yaml -f service-nodeport.yaml
$ kubectl -n meditrack get pods,svc
$ curl http://localhost:30080        # works on the box

  6. Open it from your browser

    Visit http://<EC2_PUBLIC_IP>:30080. Same golden rule: if the box's own curl succeeds but the browser can't connect, open 30080 in the security group.

✎ Which path should you pick?

Learning Docker & want the app up fast? Path A. Practising the actual Kubernetes YAML you'll use at work? Path B. Doing both on the same box, back to back, is the single best way to feel the difference between "run a container" and "run a cluster."

▲ One EC2 is for learning, not production

A single node means a single point of failure — if the box dies, everything dies, and self-healing can only reschedule onto the same node. That's completely fine for practice. Real workloads use a managed multi-node cluster (on AWS that's EKS), where the manifests you just wrote work unchanged — you'd only swap the NodePort for a proper LoadBalancer or Ingress.

◆ The whole handbook in one breath

Docker packages your app into an image and pushes it to a registry. Multi-stage keeps that image small. Compose runs the set locally. Kubernetes pulls the image and runs it as self-healing, scalable Pods described in a handful of YAML files — which you can practise end-to-end on a single EC2 box today. Docker is the workshop; Kubernetes is the stage; the registry is the door between them.

Master the workshop, and the stage stops being scary. ✎
