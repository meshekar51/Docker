TOPIC 01  ·  DOCKER

Docker Fundamentals

Objective — Understand what containers are, how Docker images work, write a Dockerfile, and use core CLI commands confidently.

1.1 What is a Container?
💡 Layman's Explanation
Imagine shipping a physical product. Before containers, you'd hand-pack each item and hope the destination has the right shelves and labels. A shipping container standardises everything — same box, same crane, any ship. Docker containers do the same for software: your app + its dependencies + its runtime, all in one portable, self-contained unit that runs the same way on your laptop, a CI server, or AWS.

A container is a lightweight, isolated process that runs on a shared OS kernel. Unlike a Virtual Machine (VM), it does not virtualise hardware — it uses Linux kernel features called namespaces (isolation) and cgroups (resource limits) to create the illusion of a private OS.

DIAGRAM 01 — Virtual Machine vs Container architecture
PropertyVirtual MachineContainer
Boot time30 – 60 seconds< 1 second
SizeGigabytes (full OS)Megabytes (libs only)
OSFull Guest OS per VMShared host kernel
IsolationFull hardware isolationProcess-level (namespaces)
PortabilityHypervisor-specificAny Docker host
1.2 Images & Layers

A Docker image is a read-only template used to create containers. Every image is built from layers — each instruction in a Dockerfile adds a new layer. Layers are cached and shared across images, making builds fast and storage efficient.

DIAGRAM 02 — Docker image layer stack (node:18 → your app)
⚡ Cache Tip
Docker rebuilds only layers that changed and all layers above them. Always COPY package.json and run npm install before copying your source code — that way, a code-only change won't re-download all dependencies.
1.3 Writing Your First Dockerfile

A Dockerfile is a plain-text recipe that tells Docker how to build your image. Every line is an instruction. Here's a production-style Dockerfile for a Node.js API:

Dockerfile Dockerfile 
# ── Stage 1: use an official Node.js LTS image as the base ──────────
FROM node:18-alpine               # Alpine = tiny (~50 MB vs 900 MB full)

# ── Set the working directory inside the container ──────────────────
WORKDIR /app

# ── Copy dependency manifests first (layer cache trick) ─────────────
COPY package*.json ./

# ── Install production dependencies only ────────────────────────────
RUN npm ci --only=production

# ── Copy the rest of the application source code ────────────────────
COPY . .

# ── Document which port the app listens on (informational) ──────────
EXPOSE 3000

# ── Default command to run when the container starts ────────────────
CMD ["node", "server.js"]
Terminal — Build & run Bash 
# Build the image (tag it myapi:1.0)
docker build -t myapi:1.0 .

# Run a container from it, map host port 8080 → container port 3000
docker run -d -p 8080:3000 --name api-server myapi:1.0

# Check it's running
curl http://localhost:8080/health
InstructionPurposeExample
FROMBase image to start fromFROM node:18-alpine
WORKDIRSet working directoryWORKDIR /app
COPYCopy files from host into imageCOPY . .
RUNExecute a shell command (build time)RUN npm install
EXPOSEDocument port (does not publish)EXPOSE 3000
ENVSet environment variableENV NODE_ENV=production
CMDDefault command at container startCMD ["node","app.js"]
ENTRYPOINTFixed command (CMD becomes args)ENTRYPOINT ["node"]

🌍 Real-World Usage
In a CI/CD pipeline (GitHub Actions, GitLab CI), the typical flow is: docker build → docker push to ECR (AWS) or GCR (GCP), then the deployment system pulls and runs the new image. You never SSH into a server to deploy — you push an image.

Networking, Volumes & Compose

Objective — Connect containers together over networks, persist data with volumes, and manage multi-container apps with Docker Compose.

2.1 Container Networking

Docker gives every container a virtual network interface. By default, containers on the same bridge network can reach each other by container name. Containers are isolated from the host unless you explicitly map ports with -p.

DIAGRAM 03 — Docker bridge network: API + Database + external traffic
Network TypeUse CaseContainer Discovery
bridge (default)Single-host developmentBy container name
hostHigh-performance, no NATShares host network stack
noneFully isolated workloadNo networking
overlayMulti-host (Docker Swarm / K8s)Across nodes via VXLAN
2.2 Volumes & Persistent Data

Containers are ephemeral — when removed, all data written inside is lost. Volumes solve this by mounting external storage into the container's filesystem.

TypeSyntaxBest For
Named volume-v pg-data:/var/lib/postgresqlDatabases — Docker manages location
Bind mount-v $(pwd)/src:/app/srcDev hot-reload — mirrors host folder
tmpfs mount--tmpfs /tmpSensitive data — RAM only, never on disk
Terminal — Named volume example Bash 
# Create a named volume
docker volume create pg-data

# Run Postgres with that volume — data survives container removal
docker run -d \
  --name postgres-db \
  -e POSTGRES_PASSWORD=secret \
  -v pg-data:/var/lib/postgresql/data \
  postgres:15

# List volumes
docker volume ls

# Inspect a volume (see where Docker stores it on the host)
docker volume inspect pg-data
2.3 Docker Compose

Running multiple docker run commands is error-prone. Docker Compose lets you define your entire stack in a single YAML file and bring it up with one command.

docker-compose.yml YAML 
version: "3.9"

services:

  api:
    build: .                          # build from local Dockerfile
    ports:
      - "8080:3000"
    environment:
      - NODE_ENV=production
      - DB_HOST=postgres-db           # service name = DNS hostname
      - DB_PORT=5432
    depends_on:
      - postgres-db                   # start DB first
    networks:
      - app-net

  postgres-db:
    image: postgres:15
    environment:
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=appdb
    volumes:
      - pg-data:/var/lib/postgresql/data
    networks:
      - app-net

volumes:
  pg-data:                             # named volume definition

networks:
  app-net:
    driver: bridge
Terminal — Compose commands Bash 
docker compose up -d            # start all services in background
docker compose logs -f api       # stream logs from api service
docker compose ps                # list running services
docker compose down              # stop & remove containers (keep volumes)
docker compose down -v          # also delete volumes (clean slate)
TOPIC 03  ·  KUBERNETES

Kubernetes Core Concepts

Objective — Understand K8s cluster architecture, deploy apps using Pods and Deployments, expose them with Services, and use kubectl effectively.

💡 Layman's Explanation
Docker Compose is great for one machine. But what if you need your app to run on 50 machines, auto-heal when a container crashes, and roll out updates without downtime? That's what Kubernetes (K8s) does — it's a container orchestrator. Think of K8s as an airline operations centre: it knows about all the planes (nodes), decides where to route passengers (containers), and automatically rebooks flights when a plane breaks (restarts failed pods).
3.1 Cluster Architecture

A Kubernetes cluster consists of a control plane (the brain) and one or more worker nodes (where containers actually run).

DIAGRAM 04 — Kubernetes cluster: Control Plane + Worker Nodes
ComponentLives OnRole
API ServerControl PlaneAll kubectl commands go here; central REST gateway
etcdControl PlaneDistributed DB storing all cluster state
SchedulerControl PlanePicks which node a new Pod runs on
Controller ManagerControl PlaneReconciles desired vs actual state (restarts failed pods)
kubeletEvery worker nodeAgent that starts/stops containers per Pod spec
kube-proxyEvery worker nodeHandles network rules for Service routing
3.2 Pods & Deployments

A Pod is the smallest deployable unit in K8s — it wraps one or more containers that share a network and storage. You rarely create Pods directly; instead you use a Deployment, which manages a ReplicaSet that keeps your desired number of Pod replicas running.

deployment.yaml YAML 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
  namespace: production
spec:
  replicas: 3                      # keep 3 Pods running at all times
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate             # zero-downtime deploys
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:                         # Pod template
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: 123456789.dkr.ecr.eu-west-1.amazonaws.com/myapi:1.0
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "100m"          # 0.1 CPU core guaranteed
              memory: "128Mi"
            limits:
              cpu: "500m"          # max 0.5 CPU core
              memory: "256Mi"
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
🔑 Key Insight — Labels & Selectors
Labels are key-value pairs attached to resources. Selectors find resources by label. This is how a Deployment finds its Pods, and how a Service finds its target Pods — decoupled by design. Change a label, and the Service instantly re-routes traffic to the new set.
3.3 Services & Networking

Pods are temporary — they get killed and recreated with new IP addresses. A Service gives your Pods a stable DNS name and IP. It load-balances traffic across all healthy Pods matching its selector.

DIAGRAM 05 — Kubernetes Service types and traffic flow
service.yaml YAML 
apiVersion: v1
kind: Service
metadata:
  name: api-service
spec:
  type: LoadBalancer              # provisions an AWS ELB automatically
  selector:
    app: api                       # routes to Pods with this label
  ports:
    - protocol: TCP
      port: 80                      # external port
      targetPort: 3000             # Pod port

# 2. Watch pods come up
kubectl get pods -w

kubectl rollout status deployment/api-deployment
🌍 Real-World Pattern
In production teams you never run kubectl set image manually. You update the image tag in deployment.yaml, commit it, and a GitOps tool like ArgoCD or Flux automatically reconciles the cluster to match the repo — complete audit trail and rollback built in.
