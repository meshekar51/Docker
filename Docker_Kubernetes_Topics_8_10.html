TOPIC 08  ·  KUBERNETES

Resource Management & Autoscaling

Objective — Prevent noisy-neighbour resource starvation, enforce namespace budgets, and build a cluster that scales Pods and Nodes automatically in response to real demand.

💡 Layman's Explanation
Without resource limits a single runaway Pod can eat all cluster CPU and starve every other app — like one table in a restaurant ordering everything on the menu. Requests are the minimum you reserve (your table booking), Limits are the maximum you're allowed (the menu cap). HPA is a waiter who opens more tables when the restaurant fills up, and Cluster Autoscaler is the manager who calls in extra staff and opens a new room when even the extra tables are full.
8.1 Requests, Limits & LimitRanges

Every container should declare requests (guaranteed resources, used by the scheduler) and limits (hard caps enforced by the kernel). A container that exceeds its memory limit is OOM-killed; one that exceeds CPU is throttled.

DIAGRAM 01 — Requests vs Limits: scheduler packing and kernel enforcement on a Node
QoS ClassConditionEviction PriorityUse For
Guaranteedrequests == limits for all containersLast evictedDatabases, critical services
Burstablerequests < limits, or only requests setMiddleMost web apps & APIs
BestEffortNo requests or limits setFirst evictedAvoid in production

A LimitRange sets per-Pod/container defaults and maxima so developers can't accidentally deploy without limits:

limitrange.yamlYAML
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
    - type: Container
      default:             # applied when limits not specified
        cpu:    "500m"
        memory: "256Mi"
      defaultRequest:      # applied when requests not specified
        cpu:    "100m"
        memory: "128Mi"
      max:                 # hard ceiling per container
        cpu:    "2"
        memory: "2Gi"
      min:
        cpu:    "50m"
        memory: "64Mi"
8.2 ResourceQuotas per Namespace

A ResourceQuota caps the total resources a namespace can consume — preventing one team's workloads from monopolising the cluster.

resourcequota.yamlYAML
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-alpha
spec:
  hard:
    # Compute
    requests.cpu:    "4"         # sum of all Pod requests
    requests.memory: "8Gi"
    limits.cpu:      "8"
    limits.memory:   "16Gi"
    # Object counts
    pods:                  "20"
    services:              "10"
    persistentvolumeclaims: "5"
    secrets:               "20"
Terminal — check quota usageBash
kubectl describe resourcequota team-quota -n team-alpha

# Output:
Name:            team-quota
Resource         Used    Hard
--------         ----    ----
limits.cpu       3200m   8
limits.memory    6Gi     16Gi
pods             12      20
requests.cpu     1600m   4
requests.memory  3Gi     8Gi
8.3 Horizontal Pod Autoscaler (HPA)

The HPA watches a metric (CPU, memory, or custom) and automatically adjusts the number of Pod replicas to keep the metric at a target. It queries the Metrics Server every 15 seconds.

DIAGRAM 02 — HPA control loop: metric → calculation → scale Deployment
hpa.yaml — CPU + memory + custom metricYAML
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-deployment
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource               # scale on CPU
      resource:
        name: cpu
        target:
          type:               Utilization
          averageUtilization: 50   # 50% of requests.cpu
    - type: Resource               # scale on memory
      resource:
        name: memory
        target:
          type:               Utilization
          averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5m before scaling down
    scaleUp:
      stabilizationWindowSeconds: 30   # react quickly to spikes
⚡ HPA Requires Requests to Be Set
HPA calculates utilisation as currentUsage / requests. If a container has no requests.cpu set, HPA cannot calculate utilisation and will refuse to scale. This is another reason LimitRanges (section 8.1) are essential — they ensure defaults exist.
8.4 Vertical Pod Autoscaler (VPA)

While HPA adds more Pods, VPA adjusts the CPU/memory requests of existing Pods over time based on observed usage. It answers: "are my resource requests even right?"

↔ HPA — Scale Out
  • Adds/removes Pod replicas
  • Best for stateless workloads
  • Responds to traffic spikes
  • Works with Deployments
  • Can combine with VPA (carefully)
↕ VPA — Scale Up/Down
  • Adjusts requests/limits of Pods
  • Best for setting right-sized requests
  • Needs Pod restart to apply (usually)
  • Works with StatefulSets too
  • Use in Off mode to just get recommendations
vpa.yaml — recommendation mode (safe in production)YAML
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: api-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind:       Deployment
    name:       api-deployment
  updatePolicy:
    updateMode: "Off"      # only recommend — never restart Pods
    # Other modes: "Initial" (set on creation), "Auto" (evict + resize)
Terminal — read VPA recommendationsBash
kubectl describe vpa api-vpa

# Output (Recommendation section):
  Recommendation:
    Container Recommendations:
      Container Name: api
      Lower Bound:   cpu: 50m    memory: 92Mi
      Target:        cpu: 200m   memory: 210Mi   ← use these
      Upper Bound:   cpu: 800m   memory: 500Mi
8.5 Cluster Autoscaler & Karpenter

HPA adds Pods but if no Node has capacity, the Pods stay Pending. The Cluster Autoscaler (or Karpenter on AWS) watches for pending Pods and provisions new Nodes — and removes underutilised Nodes to save cost.

DIAGRAM 03 — Karpenter node provisioning: Pending Pods → new EC2 Node → Pods scheduled
FeatureCluster AutoscalerKarpenter (AWS)
Provisioning speed~3-5 minutes< 60 seconds
Instance selectionPredefined node groupsAny matching instance type
Spot supportVia node groupsNative, automatic fallback
Bin-packingLimitedOptimal consolidation
Cloud supportAll major cloudsAWS only (Azure preview)
🌍 Real-World Cost Pattern
Use Karpenter with a NodePool that allows both on-demand and Spot instances. For stateless workloads (API pods), Karpenter automatically places them on Spot (70–90% cheaper). For stateful workloads (databases), label them with karpenter.sh/capacity-type: on-demand to avoid interruptions. Teams typically cut EC2 costs 50–70% this way.

Observability — Metrics, Logs & Traces

Objective — Know what's happening in your cluster at all times. Collect metrics with Prometheus, visualise with Grafana, aggregate logs with Loki, and trace requests end-to-end with OpenTelemetry.

💡 Layman's Explanation
Running a production cluster without observability is like flying a plane with no instruments. Metrics are the altitude and speed gauges (CPU, memory, request rate). Logs are the flight recorder — every event written down. Traces are the GPS track — following a single passenger's journey from check-in through security through boarding to their seat, across multiple services.

9.2 Prometheus & PromQL

Prometheus scrapes HTTP /metrics endpoints on a schedule and stores time-series data. It discovers targets automatically via Kubernetes service discovery. PromQL is its query language.

Terminal — install kube-prometheus-stack (the standard)Bash
helm repo add prometheus-community \
  https://prometheus-community.github.io/helm-charts
helm repo update

# Installs: Prometheus + Grafana + Alertmanager + kube-state-metrics
#           + node-exporter + pre-built K8s dashboards
helm install kube-prometheus \
  prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --create-namespace \
  -f prometheus-values.yaml

To expose your own app metrics, add a /metrics endpoint (use a Prometheus client library) and create a ServiceMonitor:

servicemonitor.yaml — tell Prometheus to scrape your appYAML
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name:      api-metrics
  namespace: production
  labels:
    release: kube-prometheus   # must match Prometheus selector
spec:
  selector:
    matchLabels:
      app: api
  endpoints:
    - port:     http
      path:     /metrics       # your app's metrics endpoint
      interval: 15s

Essential PromQL Queries

PromQL — production queries you'll use dailyPromQL
# CPU usage % per Pod (across all pods in production namespace)
sum(rate(container_cpu_usage_seconds_total{
  namespace="production"
}[5m])) by (pod) /
sum(kube_pod_container_resource_requests{
  resource="cpu", namespace="production"
}) by (pod) * 100

# Request error rate (5xx responses per second)
sum(rate(http_requests_total{
  status=~"5..", job="api"
}[5m]))

# P95 request latency
histogram_quantile(0.95,
  sum(rate(http_request_duration_seconds_bucket{
    job="api"
  }[5m])) by (le)
)

# Pods not running
kube_pod_status_phase{phase!~"Running|Succeeded"} == 1

# Memory usage vs limits (OOM risk indicator)
container_memory_working_set_bytes /
kube_pod_container_resource_limits{resource="memory"} * 100
9.3 Grafana Dashboards & Alertmanager

Grafana connects to Prometheus (and Loki, Tempo) as data sources and renders dashboards. The kube-prometheus-stack Helm chart ships with pre-built dashboards covering node resources, Pod states, and K8s API server metrics.

⚡ Import Community Dashboards
Visit grafana.com/dashboards and import by ID. Essential ones: 315 (Kubernetes cluster overview), 6417 (Pod resources), 13332 (Node Exporter full), 14584 (NGINX Ingress). Import by going to Dashboards → Import → paste the ID.

Define alerting rules with PrometheusRule — Alertmanager routes them to Slack, PagerDuty, email, etc.:

prometheusrule.yaml — production alertsYAML
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: api-alerts
  labels:
    release: kube-prometheus
spec:
  groups:
    - name: api.rules
      rules:
        - alert: HighErrorRate
          expr: |
            sum(rate(http_requests_total{status=~"5..",job="api"}[5m]))
            / sum(rate(http_requests_total{job="api"}[5m])) > 0.05
          for: 2m
          labels:
            severity: critical
          annotations:
            summary: "API error rate > 5% for 2 minutes"

        - alert: PodCrashLooping
          expr: increase(kube_pod_container_status_restarts_total[15m]) > 3
          for: 5m
          labels:
            severity: warning
9.4 Loki — Log Aggregation

Loki is Prometheus for logs. Unlike Elasticsearch, it indexes only labels (not full text) — making it dramatically cheaper. Promtail (or the newer Grafana Alloy) runs as a DaemonSet on every Node, tails container logs, and ships them to Loki.

DIAGRAM 05 — Loki log pipeline: Pod stdout → Promtail → Loki → Grafana
LogQL — useful log queries in GrafanaLogQL
# All errors from the api app
{app="api", namespace="production"} |= "error"

# Parse JSON and filter by HTTP status
{app="api"} | json | status >= 500

# Count error lines per minute (metric from logs)
sum(rate({app="api"} |= "error" [1m]))

# Find slow requests (>1s) from structured logs
{app="api"} | json | duration_ms > 1000
  | line_format "{{.method}} {{.path}} {{.duration_ms}}ms"
⚡ Always Log JSON
Plain text logs are nearly useless in Loki. Structure your app's logs as JSON — every field becomes a filterable label. In Node.js use pino, in Python use structlog, in Go use zap or slog. Minimum fields: level, msg, timestamp, requestId, statusCode, durationMs.
9.5 Distributed Tracing & OpenTelemetry

A single user request may touch 5+ services. Distributed tracing follows it end-to-end by propagating a trace-id through HTTP headers and collecting spans — each representing work done in one service. OpenTelemetry (OTel) is the vendor-neutral standard for instrumentation.

otel-collector.yaml — central collection & routingYAML
# OpenTelemetry Collector deployed as a Deployment in the cluster
# Receives spans from apps, processes them, exports to Tempo
config:
  receivers:
    otlp:                        # standard OTLP protocol
      protocols:
        grpc: # port 4317
        http: # port 4318

  processors:
    batch:                        # buffer spans before export
    memory_limiter:
      limit_mib: 400
    k8sattributes:               # auto-attach pod/node labels to spans

  exporters:
    otlp/tempo:
      endpoint: tempo:4317

  service:
    pipelines:
      traces:
        receivers:  [otlp]
        processors: [memory_limiter, k8sattributes, batch]
        exporters:  [otlp/tempo]
ToolRoleQuery Language
PrometheusMetrics storage & evaluationPromQL
GrafanaDashboards, unified UI for all three pillarsPromQL / LogQL / TraceQL
LokiLog aggregation (label-indexed)LogQL
TempoDistributed trace storageTraceQL
AlertmanagerRoute alerts → Slack, PagerDuty, email—
OpenTelemetryVendor-neutral SDK + collectorOTLP protocol
🌍 The Grafana Stack on AWS
The grafana/lgtm Helm stack (Loki + Grafana + Tempo + Mimir) is the modern open-source answer to Datadog. For production, configure Loki and Tempo to use S3 as the object storage backend — zero EC2 disk for log/trace storage, infinite retention, and pennies per GB. A cluster handling 10k req/s typically costs $20–40/month in S3 storage for 30-day retention.

RBAC & Network Security

Objective — Apply least-privilege access control so humans and services can only do what they need, and isolate workloads at the network level so a compromised Pod cannot reach everything.

💡 Layman's Explanation
An office building has key cards (RBAC) — the cleaner can open the cleaning cupboard but not the server room. The building also has fire doors (Network Policies) — even if someone bypasses the key card, they can't get from reception to the data centre without going through another locked door. Kubernetes needs both: access control for the API (RBAC) and network-level isolation between Pods (NetworkPolicy).
10.1 RBAC — Roles & Bindings
DIAGRAM 06 — RBAC: Subject → Binding → Role → Resources (verbs)
rbac.yaml — read-only developer role in productionYAML
# ── Role: read-only access to Pods and logs in one namespace ──────
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name:      dev-readonly
  namespace: production
rules:
  - apiGroups: [""]
    resources: [pods, pods/log, services, events]
    verbs:     [get, list, watch]
  - apiGroups: [apps]
    resources: [deployments, replicasets]
    verbs:     [get, list, watch]
---
# ── RoleBinding: bind the role to a user ──────────────────────────
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name:      dev-readonly-binding
  namespace: production
subjects:
  - kind: User
    name: alice@company.com
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind:     Role
  name:     dev-readonly
  apiGroup: rbac.authorization.k8s.io
Terminal — test and audit RBACBash
# Can alice list pods in production?
kubectl auth can-i list pods \
  --namespace production \
  --as alice@company.com
# yes

# Can alice delete deployments?
kubectl auth can-i delete deployments \
  --namespace production \
  --as alice@company.com
# no

# What can the api ServiceAccount do?
kubectl auth can-i --list \
  --namespace production \
  --as system:serviceaccount:production:api-sa
10.2 ServiceAccounts & IRSA (AWS)

Every Pod runs as a ServiceAccount. By default it uses the default ServiceAccount which has no permissions (but still gets a mounted token). For AWS, IRSA (IAM Roles for Service Accounts) lets a Pod assume an IAM role without any long-lived credentials in the cluster.

serviceaccount-irsa.yaml — Pod ↔ AWS S3 without secretsYAML
# 1. Create a ServiceAccount annotated with the IAM role ARN
apiVersion: v1
kind: ServiceAccount
metadata:
  name:      api-sa
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn:
      arn:aws:iam::123456789:role/api-s3-reader
---
# 2. Reference the ServiceAccount in the Deployment
spec:
  template:
    spec:
      serviceAccountName: api-sa   # Pod gets AWS credentials automatically
      containers:
        - name: api
          # No AWS_ACCESS_KEY_ID or AWS_SECRET_ACCESS_KEY needed!
          # AWS SDK auto-discovers credentials via OIDC token
10.3 Network Policies

By default, all Pods in a cluster can talk to all other Pods. NetworkPolicies act as a firewall — restricting which Pods can send or receive traffic. They require a CNI plugin that supports them (Calico, Cilium, Weave). The AWS VPC CNI does not support NetworkPolicy — use Cilium or the AWS Network Policy Controller add-on.

networkpolicy.yaml — postgres only accepts from api podsYAML
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name:      postgres-ingress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: postgres             # this policy applies TO postgres pods
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api           # ONLY api pods may connect
      ports:
        - port: 5432
---
# Default-deny all ingress in the namespace (add first, then allow)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name:      default-deny-ingress
  namespace: production
spec:
  podSelector: {}              # matches ALL pods
  policyTypes: [Ingress]
  # no ingress rules = deny everything by default
10.4 Pod Security Standards

Pod Security Standards (PSS) replaced PodSecurityPolicies in K8s 1.25. They define three profiles applied at the namespace level via labels:

ProfileWhat it blocksEnforcement label
privilegedNothing — unrestricted (system namespaces only)pod-security.kubernetes.io/enforce: privileged
baselineMost known privilege escalations (no host network/PID, no privileged containers)pod-security.kubernetes.io/enforce: baseline
restrictedAll of baseline + must run as non-root, drop all capabilities, no /proc writespod-security.kubernetes.io/enforce: restricted
namespace.yaml — enforce restricted PSS on production namespaceYAML
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted   # block non-compliant pods
    pod-security.kubernetes.io/warn:    restricted   # warn on kubectl apply
    pod-security.kubernetes.io/audit:   restricted   # log violations
10.5 Policy Enforcement with Kyverno

Kyverno is a Kubernetes-native policy engine. It runs as an admission controller — intercepting every resource create/update and either validating, mutating, or generating resources based on policies written as YAML.

V
Validate — block non-compliant resources
Reject Pods that don't have resource limits, reject images from non-approved registries, require specific labels on all Deployments.
M
Mutate — auto-fix resources on admission
Automatically add default labels, inject sidecars, set imagePullPolicy: Always when :latest tag is used.
G
Generate — create resources automatically
When a new namespace is created, automatically create a default NetworkPolicy, LimitRange, and ResourceQuota inside it.
kyverno-policy.yaml — require resource limits + approved registryYAML
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-limits-and-registry
spec:
  validationFailureAction: Enforce     # block non-compliant (vs Audit)
  rules:
    - name: require-resource-limits
      match:
        resources:
          kinds: [Pod]
      validate:
        message: "CPU and memory limits are required on all containers"
        pattern:
          spec:
            containers:
              - resources:
                  limits:
                    cpu:    "?*"
                    memory: "?*"

    - name: approved-registry-only
      match:
        resources:
          kinds: [Pod]
          namespaces: [production]
      validate:
        message: "Images must come from our ECR registry"
        pattern:
          spec:
            containers:
              - image: "123456789.dkr.ecr.eu-west-1.amazonaws.com/*"
🔐 Security Layering — Defence in Depth
No single control is sufficient. Production clusters should layer all of: RBAC (who can call the API), NetworkPolicy (which pods can communicate), Pod Security Standards (what Pods are allowed to do), Kyverno / OPA (policy as code, enforced at admission), and image scanning (Trivy in CI, ECR scanning at push). Assume each layer will be bypassed individually and design so breaching one still doesn't give an attacker everything.
