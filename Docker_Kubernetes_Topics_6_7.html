TOPIC 06  ·  KUBERNETES

Ingress Controllers & TLS

Objective — Stop creating one cloud load balancer per service. Use a single Ingress Controller to route HTTP/HTTPS traffic to many services, and automate TLS certificates with cert-manager.

💡 Layman's Explanation
A hotel has one front door, not a separate door for the restaurant, gym, and spa. A LoadBalancer Service is like giving each service its own front door (and paying for each door separately — at AWS prices). An Ingress Controller is the hotel's reception desk: one public IP, and it routes guests to the right place based on their request. api.myapp.com → API pods, myapp.com/docs → docs pods. And cert-manager is the automatic locksmith that keeps the front door's lock (TLS cert) fresh — no manual renewal ever.
6.1 The Problem with LoadBalancer Services

From Topic 03, a type: LoadBalancer Service provisions a cloud load balancer (AWS ELB, GCP LB) per service. This has real costs:

ProblemDetail
CostEach AWS ELB costs ~$18/month + data transfer. 10 services = $180+/month in LB fees alone
IP sprawlEvery service gets a different external IP — hard to manage DNS, firewalls, and certificates
No HTTP routingLoadBalancer operates at L4 (TCP) — it cannot route based on URL path or hostname
No TLS terminationEach service must manage its own TLS certificate independently
6.2 What is an Ingress Controller?

An Ingress Controller is a Pod running inside your cluster (typically NGINX or Traefik) exposed via a single LoadBalancer Service. An Ingress resource then defines routing rules — the controller reads these rules and configures itself dynamically.

DIAGRAM 01 — Ingress Controller: one LB, many services, host & path routing
1
Install NGINX Ingress Controller
Deploys the controller pods and creates one LoadBalancer Service. All other services become ClusterIP — internal only.
2
Create Ingress resources
Each Ingress resource defines routing rules (hosts, paths → services). The controller watches for them and reloads automatically.
3
Point DNS to the LB IP
All your domains (api.myapp.com, myapp.com) point to the same single IP via DNS A records or CNAME.
4
Add TLS via cert-manager
cert-manager handles ACME challenges with Let's Encrypt, stores certs as K8s Secrets, and auto-renews before expiry.
Terminal — Install NGINX Ingress Controller via Helm Bash 
# Add the official ingress-nginx Helm repository
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Install into its own namespace
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.replicaCount=2

# Get the external IP assigned to the controller
kubectl get svc -n ingress-nginx

# Output:
NAME                       TYPE           CLUSTER-IP    EXTERNAL-IP       PORT(S)
ingress-nginx-controller   LoadBalancer   10.100.0.10   54.123.45.67      80,443
6.3 Routing Rules — Host & Path

An Ingress resource is a set of rules. Each rule matches traffic by hostname and/or URL path and forwards it to a backend Service.

ingress.yaml — host-based + path-based routing YAML 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  namespace: production
  annotations:
    kubernetes.io/ingress.class: "nginx"
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  tls:                                  # TLS — cert stored as a Secret
    - hosts:
        - api.myapp.com
        - myapp.com
      secretName: myapp-tls            # cert-manager writes here
  rules:
    # ── Rule 1: route by hostname ─────────────────────────────────
    - host: api.myapp.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-service
                port: number: 80

    # ── Rule 2: route by URL path on same domain ──────────────────
    - host: myapp.com
      http:
        paths:
          - path: /docs
            pathType: Prefix
            backend:
              service:
                name: docs-service
                port: number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port: number: 80
pathTypeMatch behaviourExample
PrefixMatch path prefix (most common)/api matches /api, /api/users, /api/v2
ExactMatch exact path only/health matches only /health
ImplementationSpecificController decides (regex in NGINX)/api/(.*) with capture groups
6.4 TLS with cert-manager & Let's Encrypt

cert-manager is a Kubernetes add-on that automates the full lifecycle of TLS certificates: request → ACME challenge → issue → store as Secret → renew (30 days before expiry).

DIAGRAM 02 — cert-manager ACME flow: ClusterIssuer → Certificate → TLS Secret
Terminal — Install cert-manager Bash 
# Install cert-manager (includes CRDs)
helm repo add jetstack https://charts.jetstack.io
helm repo update

helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --set installCRDs=true

# Verify all pods are running
kubectl get pods -n cert-manager
clusterissuer.yaml — Let's Encrypt production issuer YAML 
apiVersion: cert-manager.io/v1
kind: ClusterIssuer                   # cluster-wide (vs Issuer: namespace-scoped)
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: ops@myapp.com              # expiry warning emails go here
    privateKeySecretRef:
      name: letsencrypt-prod-key
    solvers:
      - http01:                       # HTTP-01 challenge via Ingress
          ingress:
            class: nginx
ingress-tls.yaml — Ingress with cert-manager annotation YAML 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"  # ← triggers cert request
    kubernetes.io/ingress.class:    "nginx"
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  tls:
    - hosts: [api.myapp.com]
      secretName: api-tls          # cert-manager creates this Secret
  rules:
    - host: api.myapp.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-service
                port: number: 80
⚡ Staging First
Let's Encrypt production has rate limits (5 certs per domain per week). Always test with the staging server first: https://acme-staging-v02.api.letsencrypt.org/directory. Create a letsencrypt-staging ClusterIssuer, verify it works (you'll get an untrusted cert), then switch to production.
6.5 Useful NGINX Ingress Annotations
AnnotationEffect
ssl-redirect: "true"Redirect all HTTP → HTTPS automatically
proxy-body-size: "50m"Allow request bodies up to 50 MB (file uploads)
proxy-read-timeout: "120"Increase timeout for slow upstream (default 60s)
rate-limit: "10r/s"Rate-limit to 10 requests/second per IP
whitelist-source-rangeRestrict access to specific IP CIDRs (e.g. office VPN only)
rewrite-target: /Strip path prefix before forwarding (e.g. /api/v1 → /)
use-gzip: "true"Enable gzip compression on responses
enable-cors: "true"Add CORS headers for browser-side API calls

Helm — Kubernetes Package Manager

Objective — Stop copy-pasting YAML. Use Helm to template, version, and share Kubernetes applications — then package your own full-stack app as a chart.

💡 Layman's Explanation
Installing Postgres on Kubernetes without Helm requires writing ~200 lines of YAML across 6 files, all by hand, getting the labels right, the Service names consistent, and the volume claims correct. With Helm it's helm install my-db bitnami/postgresql --set auth.password=secret — one command. Helm is to Kubernetes what apt or npm is to Linux / Node: a package manager that bundles everything an app needs into one installable, upgradeable, rollback-able unit called a Chart.
7.1 Why Helm?
❌ Raw YAML approach
  • Duplicate values across many files (image tag in 3 places)
  • No easy way to diff dev vs prod config
  • Upgrade = kubectl apply and hope nothing breaks
  • Rollback = manually re-apply old YAML
  • Sharing requires copying & editing files
  • No versioning or release history
✅ Helm approach
  • Single values.yaml controls the whole app
  • Override values per environment with -f prod.yaml
  • helm upgrade does atomic, tracked upgrades
  • helm rollback in one command with history
  • Share via chart repositories (Artifact Hub)
  • Full release history: helm history my-release
7.2 Chart Structure

A Chart is a directory with a specific layout. Run helm create myapp to scaffold it:

myapp/ ├── Chart.yaml # chart metadata (name, version, appVersion) ├── values.yaml # default values — users override these ├── templates/ # Go-templated K8s manifests │ ├── deployment.yaml # uses {{ .Values.image.tag }} etc. │ ├── service.yaml │ ├── ingress.yaml │ ├── configmap.yaml │ ├── secret.yaml │ ├── _helpers.tpl # reusable template snippets │ └── NOTES.txt # printed after helm install ├── charts/ # sub-charts (dependencies) └── .helmignore # like .gitignore for helm package
Chart.yaml YAML 
apiVersion: v2
name: myapp
description: Node.js API + Postgres full-stack chart
type: application
version: 1.3.0          # chart version (bumped when chart changes)
appVersion: "2.1.0"    # your app version (informational)
dependencies:
  - name:       postgresql
    version:    "13.x.x"
    repository: https://charts.bitnami.com/bitnami
    condition:  postgresql.enabled    # skip if using external DB
7.3 Values & Templates

Templates use Go's text/template syntax. Values flow in from values.yaml (or overrides) and are referenced as {{ .Values.key }}.

values.yaml — default values for the chart YAML 
replicaCount: 2

image:
  repository: 123456789.dkr.ecr.eu-west-1.amazonaws.com/myapi
  tag: "2.1.0"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

ingress:
  enabled: true
  host: api.myapp.com
  tls:
    enabled: true
    secretName: api-tls
    clusterIssuer: letsencrypt-prod

resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"

postgresql:
  enabled: true             # set false to use external RDS
  auth:
    database: appdb
    existingSecret: db-credentials
templates/deployment.yaml — Go template consuming values YAML 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}    # from _helpers.tpl
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}     # ← from values.yaml
  selector:
    matchLabels:
      {{- include "myapp.selectorLabels" . | nindent 6 }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          resources:
            {{- toYaml .Values.resources | nindent 12 }}

# ── Conditional block — only render if ingress.enabled == true ────
# (in templates/ingress.yaml)
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    {{- if .Values.ingress.tls.enabled }}
    cert-manager.io/cluster-issuer: {{ .Values.ingress.tls.clusterIssuer | quote }}
    {{- end }}
{{- end }}
templates/_helpers.tpl — reusable named templates Go Template 
{{/*
Expand the name of the chart.
*/}}
{{- define "myapp.fullname" -}}
{{- .Release.Name }}-{{ .Chart.Name }}
{{- end }}

{{/*
Common labels applied to all resources
*/}}
{{- define "myapp.labels" -}}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}
7.4 Core Helm Commands
Terminal — day-to-day Helm workflow Bash 
# ── Repository management ─────────────────────────────────────────
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update                          # refresh chart index
helm search repo postgresql              # find charts in repos
helm show values bitnami/postgresql      # view all default values

# ── Install ───────────────────────────────────────────────────────
helm install my-api ./myapp \            # from local chart
  --namespace production \
  --create-namespace \
  -f values-prod.yaml \                  # override with prod values
  --set image.tag="2.1.0"              # override single value

# ── Upgrade (deploy new version) ──────────────────────────────────
helm upgrade my-api ./myapp \
  -f values-prod.yaml \
  --set image.tag="2.2.0" \
  --atomic                              # auto-rollback if upgrade fails

# ── Inspect releases ──────────────────────────────────────────────
helm list -n production                 # list all releases in namespace
helm history my-api                     # show revision history
helm status my-api                      # current status + NOTES

# ── Rollback ──────────────────────────────────────────────────────
helm rollback my-api 2                  # roll back to revision 2

# ── Dry run / debug ───────────────────────────────────────────────
helm template my-api ./myapp \          # render templates without installing
  -f values-prod.yaml | kubectl apply --dry-run=client -f -

# ── Uninstall ─────────────────────────────────────────────────────
helm uninstall my-api -n production    # removes all resources

7.5 Packaging Your Own Chart
Terminal — create, lint, package, push Bash 
# 1. Scaffold a new chart
helm create myapp

# 2. Pull in dependencies declared in Chart.yaml
helm dependency update ./myapp

# 3. Lint — catches YAML errors and best-practice violations
helm lint ./myapp -f values-prod.yaml

# 4. Render locally to review generated YAML
helm template my-api ./myapp -f values-prod.yaml

# 5. Package into a versioned tarball
helm package ./myapp
# produces: myapp-1.3.0.tgz

# 6. Push to OCI registry (e.g. AWS ECR, GitHub Packages)
helm push myapp-1.3.0.tgz oci://123456789.dkr.ecr.eu-west-1.amazonaws.com/charts

# 7. Install from registry in CI/CD
helm install my-api \
  oci://123456789.dkr.ecr.eu-west-1.amazonaws.com/charts/myapp \
  --version 1.3.0 \
  -f values-prod.yaml

Create environment-specific override files to keep your values.yaml minimal and clean:

values-prod.yaml — production overrides only YAML 
# Only override what differs from defaults in values.yaml
replicaCount: 5

image:
  tag: "2.1.0"

ingress:
  host: api.myapp.com
  tls:
    clusterIssuer: letsencrypt-prod

resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "2"
    memory: "1Gi"

postgresql:
  enabled: false           # use AWS RDS in production
7.6 Helmfile — Managing Multiple Charts

When you have multiple charts to deploy together (API + monitoring + ingress-controller + cert-manager), Helmfile acts as a declarative wrapper — one file describes your entire cluster's desired state.

helmfile.yaml — full cluster in one declarative file YAML 
repositories:
  - name: ingress-nginx
    url:  https://kubernetes.github.io/ingress-nginx
  - name: jetstack
    url:  https://charts.jetstack.io

releases:
  # ── Infrastructure layer ─────────────────────────────────────────
  - name: ingress-nginx
    chart: ingress-nginx/ingress-nginx
    namespace: ingress-nginx
    createNamespace: true
    values:
      - controller.replicaCount: 2

  - name: cert-manager
    chart: jetstack/cert-manager
    namespace: cert-manager
    createNamespace: true
    set:
      - name:  installCRDs
        value: "true"
    needs: [ingress-nginx/ingress-nginx]   # deploy in order

  # ── Application layer ─────────────────────────────────────────────
  - name: my-api
    chart: ./myapp
    namespace: production
    createNamespace: true
    values:
      - values-prod.yaml
    needs: [cert-manager/cert-manager]
Terminal — Helmfile commands Bash 
# Install Helmfile
brew install helmfile   # macOS

# Preview all changes (like terraform plan)
helmfile diff

# Deploy everything in dependency order
helmfile apply

# Deploy only specific release
helmfile apply --selector name=my-api

# Destroy everything
helmfile destroy
🌍 Real-World GitOps Pattern
Most production teams store their helmfile.yaml and all values-*.yaml files in Git. A tool like ArgoCD watches the repo and runs helmfile apply automatically when the main branch changes. This gives you: full audit trail, PR-based approval, instant rollback (revert the commit), and zero manual kubectl or helm commands in production.
