TOPIC 04  ·  KUBERNETES

ConfigMaps & Secrets

Objective — Separate configuration from your container image, inject environment-specific values at runtime, and store sensitive credentials safely.

💡 Layman's Explanation
Imagine a recipe book (your Docker image). The recipe says "add salt" but doesn't say how much — that's decided at serving time depending on the diner. ConfigMaps are the adjustable settings (salt amount = database host, log level). Secrets are like the recipe's lock box — they hold the private wine cellar code (passwords, API keys) and are handled differently so they never appear in plain text in your Git history.
4.1 What is a ConfigMap?

A ConfigMap is a Kubernetes object that stores non-sensitive configuration as key-value pairs. It decouples configuration from your container image, so the same image runs in dev, staging, and production with different settings.

❌ Without ConfigMap
  • Config baked into Dockerfile with ENV
  • Different images per environment
  • Changing config = rebuild + redeploy
  • Secrets visible in image layers
  • Can't update config without downtime
✅ With ConfigMap
  • One image, many environments
  • Config updated independently of code
  • Changes can trigger rolling restarts
  • Mounted as files or env variables
  • Auditable — tracked in Git as YAML

You can create a ConfigMap from a literal value, a file, or a YAML manifest:

configmap.yaml YAML 
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config
  namespace: production
data:
  # Simple key-value pairs — injected as env vars
  NODE_ENV: "production"
  LOG_LEVEL: "info"
  DB_HOST: "postgres-service"
  DB_PORT: "5432"
  DB_NAME: "appdb"
  MAX_CONNECTIONS: "20"

  # Multi-line value — mounted as a file
  app.properties: |
    server.port=3000
    server.timeout=30s
    cache.ttl=300
    feature.dark-mode=true
Terminal — Imperative alternatives Bash 
# From literal values
kubectl create configmap api-config \
  --from-literal=NODE_ENV=production \
  --from-literal=LOG_LEVEL=info

# From a .env file
kubectl create configmap api-config --from-env-file=.env.production

# From a file (key = filename, value = file content)
kubectl create configmap nginx-config --from-file=nginx.conf

# View ConfigMap contents
kubectl get configmap api-config -o yaml
4.2 Injecting Config into Pods

There are two ways to consume a ConfigMap inside a Pod: as environment variables or as mounted files. Both approaches can be used together.

DIAGRAM 01 — Two ways to consume a ConfigMap inside a Pod
deployment.yaml — consuming ConfigMap both ways YAML 
spec:
  containers:
    - name: api
      image: myapi:1.0

      # ── Method A: all keys as env vars ───────────────────────
      envFrom:
        - configMapRef:
            name: api-config       # every key becomes an env var

      # ── Method B: specific key as env var ────────────────────
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: api-config
              key:  LOG_LEVEL

      # ── Method C: mount ConfigMap as files ───────────────────
      volumeMounts:
        - name: config-vol
          mountPath: /etc/config    # files appear here
          readOnly: true

  volumes:
    - name: config-vol
      configMap:
        name: api-config
MethodUse WhenHot-Reload?
envFrom (all keys)Many configs, app reads env varsNo — restart required
env.valueFrom (single key)Selective injection, rename keysNo — restart required
volumeMount (files)Config files (nginx.conf, app.properties)Yes — K8s updates files live
4.3 Kubernetes Secrets

A Secret is similar to a ConfigMap but designed for sensitive data: passwords, tokens, TLS certificates, and API keys. Values are base64-encoded (not encrypted by default — see 4.4 for real security).

secret.yaml YAML 
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: production
type: Opaque                     # generic secret type
data:
  # Values MUST be base64-encoded
  # echo -n 'mysecretpass' | base64  →  bXlzZWNyZXRwYXNz
  POSTGRES_PASSWORD: bXlzZWNyZXRwYXNz   # base64("mysecretpass")
  POSTGRES_USER:     YXBpdXNlcg==        # base64("apiuser")

# ── Alternatively: use stringData (K8s encodes automatically) ──
# stringData:                   <-- NEVER commit this to Git!
#   POSTGRES_PASSWORD: mysecretpass
Terminal — Creating Secrets imperatively (preferred over YAML for sensitive data) Bash 
# From literal — secret never touches a file on disk
kubectl create secret generic db-credentials \
  --from-literal=POSTGRES_PASSWORD=mysecretpass \
  --from-literal=POSTGRES_USER=apiuser

# TLS certificate secret
kubectl create secret tls api-tls \
  --cert=tls.crt \
  --key=tls.key

# Docker registry credentials (for pulling private images)
kubectl create secret docker-registry ecr-creds \
  --docker-server=123456789.dkr.ecr.eu-west-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password)

# Decode a secret value to verify
kubectl get secret db-credentials -o jsonpath='{.data.POSTGRES_PASSWORD}' | base64 -d

Consuming a Secret in a Pod works exactly like ConfigMap — but use secretRef / secretKeyRef:

deployment.yaml — referencing a Secret YAML 
containers:
  - name: api
    env:
      - name:  DB_PASSWORD
        valueFrom:
          secretKeyRef:           # ← secretKeyRef, not configMapKeyRef
            name: db-credentials
            key:  POSTGRES_PASSWORD
      - name:  DB_USER
        valueFrom:
          secretKeyRef:
            name: db-credentials
            key:  POSTGRES_USER

    # Also combine with ConfigMap for non-sensitive config
    envFrom:
      - configMapRef:
          name: api-config
Secret TypeUse CaseAuto-created by
OpaqueGeneric passwords, tokens, API keysUser
kubernetes.io/tlsTLS certificates for Ingresskubectl / cert-manager
kubernetes.io/dockerconfigjsonPull from private registrykubectl / cloud IAM
kubernetes.io/service-account-tokenPod-to-API-server authK8s automatically
4.4 Security Best Practices
🔐 Critical Reality Check
Kubernetes Secrets are base64-encoded, not encrypted. Anyone with kubectl get secret access can decode them trivially. Base64 is an encoding — not a security measure. You must layer proper security on top.
1
Enable Encryption at Rest
Configure the K8s API server with an EncryptionConfiguration resource using AES-CBC or KMS. Secrets stored in etcd are then encrypted on disk. On managed clusters (EKS, GKE, AKS) this is often one checkbox.
2
Use an External Secret Manager
AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager store secrets with full audit logs, rotation, and fine-grained access. Tools like External Secrets Operator or Vault Agent Injector sync external secrets into K8s Secrets automatically.
3
Restrict RBAC on Secrets
Never give a service account get secrets permission cluster-wide. Grant access only to specific Secrets in specific namespaces. Audit with kubectl auth can-i get secrets --as=system:serviceaccount:default:my-sa.
4
Never Commit Secrets to Git
Use stringData Secrets only in CI pipelines, never in committed YAML. Use Sealed Secrets (Bitnami) or SOPS (Mozilla) to encrypt secret YAML before committing — only the cluster can decrypt them.
5
Mount as Files, Not Env Vars (where possible)
Environment variables are visible via docker inspect, crash dumps, and debug tools. Mounted secret files are slightly harder to leak, and some apps support live-reloading them — useful for rotating credentials without restarts.

Persistent Volumes & StatefulSets

Objective — Understand how K8s manages durable storage, provision volumes automatically using StorageClasses, and run stateful workloads like databases reliably with StatefulSets.

💡 Layman's Explanation
A standard Kubernetes Pod is like a hotel room — disposable, identical, and reset after checkout. Databases can't live in hotel rooms: they need a permanent address with storage that survives when they move. A PersistentVolume is the storage unit (like a safe-deposit box at the bank), a PersistentVolumeClaim is the key (your pod claims it), and a StatefulSet is the manager who ensures each database instance always gets the same box back — even after a crash.
5.1 The Problem with Ephemeral Storage

By default, all data written inside a container is stored in its writable layer. When the container restarts or the Pod is rescheduled to a different node, that data is gone. This is fine for stateless apps but fatal for databases.

DIAGRAM 03 — Ephemeral storage: data lost on Pod reschedule
5.2 PersistentVolumes & Claims

Kubernetes separates who provides storage from who uses it through two resources:

  • PersistentVolume (PV) — a piece of storage provisioned by an admin (or automatically). It exists in the cluster independently of any Pod. Think of it as the actual disk.
  • PersistentVolumeClaim (PVC) — a request for storage by a user. It says "I need 10 GB of fast SSD storage." K8s matches (binds) it to a suitable PV. Think of it as the request form.
DIAGRAM 04 — PersistentVolume lifecycle: provision → claim → mount → release
pvc.yaml — PersistentVolumeClaim for Postgres YAML 
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce             # one node can mount read/write at a time
  storageClassName: gp3        # AWS EBS gp3 (defined in StorageClass)
  resources:
    requests:
      storage: 20Gi
Access ModeAbbreviationMeaningTypical Use
ReadWriteOnceRWOOne node, read+writeDatabases (Postgres, MySQL)
ReadOnlyManyROXMany nodes, read onlyStatic assets, config files
ReadWriteManyRWXMany nodes, read+writeShared storage (NFS, EFS)
ReadWriteOncePodRWOPOne Pod, read+write (K8s 1.22+)Strict single-writer guarantee
5.3 StorageClasses & Dynamic Provisioning

In production you never pre-create PVs manually. Instead, a StorageClass defines a type of storage (SSD, HDD, NFS) and a provisioner that creates PVs on demand when a PVC is submitted. This is called dynamic provisioning.

storageclass.yaml — AWS EBS gp3 YAML 
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
provisioner: ebs.csi.aws.com    # AWS EBS CSI driver
volumeBindingMode: WaitForFirstConsumer   # create disk in same AZ as Pod
reclaimPolicy: Retain           # keep disk even if PVC is deleted
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
  encrypted: "true"
reclaimPolicyWhat happens when PVC is deletedUse
RetainPV stays, data preserved — admin must clean upProduction databases
DeletePV and underlying disk are deleted automaticallyDev / ephemeral workloads
RecycleDeprecated — data scrubbed, PV re-usedAvoid
⚡ Always Use Retain for Databases
If a developer accidentally runs kubectl delete pvc postgres-pvc with Delete policy, your entire database is gone — with no undo. Use Retain and add additional protection with kubectl annotate pv <name> "pv.kubernetes.io/provisioned-by=protected".
5.4 StatefulSets

A StatefulSet is like a Deployment but for stateful workloads. It provides three guarantees that a Deployment cannot:

🚀 Deployment
  • Pods get random names (api-5c9b4d-xyz)
  • Pods are interchangeable / stateless
  • All Pods share one PVC (or none)
  • Scale up/down in any order
  • Restart: any Pod on any Node
🗄️ StatefulSet
  • Pods get stable names (postgres-0, postgres-1)
  • Each Pod has its own identity + storage
  • Each Pod gets its own PVC (volumeClaimTemplates)
  • Scale in strict order (0 → 1 → 2)
  • Restart: same Pod name, same PVC, any Node
DIAGRAM 05 — StatefulSet: stable names, individual PVCs, headless Service DNS
5.5 Running PostgreSQL in Kubernetes

Let's put it all together: a production-style StatefulSet for PostgreSQL using a ConfigMap for config, a Secret for the password, and a PVC per replica via volumeClaimTemplates.

postgres-statefulset.yaml YAML 
# ── Headless Service: gives each Pod a stable DNS name ───────────────
apiVersion: v1
kind: Service
metadata:
  name: postgres-svc
spec:
  clusterIP: None                  # headless — no load-balancing
  selector:
    app: postgres
  ports:
    - port: 5432
---
# ── StatefulSet ───────────────────────────────────────────────────────
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-svc        # links to headless service above
  replicas: 2                       # primary + 1 replica
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:15
          envFrom:
            - configMapRef:
                name: postgres-config  # POSTGRES_DB, etc.
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key:  POSTGRES_PASSWORD
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
          resources:
            requests:
              cpu: "250m"
              memory: "512Mi"
            limits:
              cpu: "1"
              memory: "2Gi"

  # ── Each Pod gets its OWN PVC, auto-named data-postgres-<ordinal> ─
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: gp3
        resources:
          requests:
            storage: 20Gi
Terminal — Deploy and verify Bash 
# Apply everything
kubectl apply -f postgres-configmap.yaml
kubectl apply -f postgres-statefulset.yaml

# Watch pods start in order (0 must be Ready before 1 starts)
kubectl get pods -l app=postgres -w

# Output:
NAME          READY   STATUS    RESTARTS   AGE
postgres-0    1/1     Running   0          30s
postgres-1    1/1     Running   0          45s

# Check PVCs — each pod has its own
kubectl get pvc

# Output:
NAME               STATUS   VOLUME           CAPACITY   ACCESS MODES
data-postgres-0    Bound    pvc-abc123...    20Gi       RWO
data-postgres-1    Bound    pvc-def456...    20Gi       RWO

# Connect to postgres-0 (primary) via stable DNS
kubectl exec -it postgres-0 -- psql -U apiuser -d appdb

# Delete pod — K8s recreates it with same name + same PVC
kubectl delete pod postgres-0
kubectl get pods -l app=postgres   # postgres-0 comes back
🌍 Real-World Note
In production, most teams don't run databases directly in Kubernetes — they use managed services like AWS RDS, Google Cloud SQL, or Azure Database for the operational simplicity (backups, patching, HA). StatefulSets are valuable when you need full control, run in an air-gapped environment, or use databases not available as managed services (TimescaleDB, CockroachDB, custom Postgres with extensions).
⚠️ Deleting a StatefulSet
kubectl delete statefulset postgres does not delete the PVCs. This is intentional — data is preserved. You must explicitly run kubectl delete pvc data-postgres-0 data-postgres-1 to remove data. Always verify with kubectl get pvc after deletion.
